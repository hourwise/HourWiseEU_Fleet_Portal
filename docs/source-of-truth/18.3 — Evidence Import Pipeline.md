# Purpose

The Evidence Import Pipeline is responsible for safely receiving, validating, storing and preparing operational evidence for compliance processing.

It is the front door of the Compliance Intelligence Platform.

No evidence should reach the Timeline Engine, Compliance Engine, Atlas or Reporting Engine unless it has passed through this pipeline.

---

# Core Principle

Original evidence must always be preserved.

The platform may create derived data, metadata, parsed records and reports, but the original uploaded evidence must remain unchanged.

---

# Supported Evidence Types

## Current

* Driver Card files
* Vehicle Unit files
* HourWise helper captures
* Driver App records
* Manual uploads
* Incident photographs
* Expense receipts
* Vehicle check submissions
* Documents and certificates

## Future

* GPS data
* Telematics feeds
* OCR-extracted documents
* Workshop systems
* Payroll exports
* Accounting exports
* External compliance systems

---

# Pipeline Overview

```text
Evidence Received
        ↓
Authentication Check
        ↓
Permission Check
        ↓
Upload Validation
        ↓
File Integrity Check
        ↓
Duplicate Detection
        ↓
Metadata Extraction
        ↓
Evidence Storage
        ↓
Processing Queue
        ↓
Parser / Normaliser
        ↓
Timeline Engine
        ↓
Compliance Engine
        ↓
Reporting / Atlas
```

---

# Pipeline Responsibilities

The import pipeline must:

* receive evidence
* validate uploads
* reject unsafe files
* preserve originals
* identify evidence type
* extract metadata
* detect duplicates
* assign ownership
* record audit events
* track processing status
* allow retry
* report failures clearly

---

# Import Sources

## Browser Upload

Used mainly in the Fleet Portal.

Examples:

* manual driver card upload
* manual VU upload
* document upload
* receipt upload
* incident evidence upload

---

## Driver App Sync

Used when driver app records are synced to the platform.

Examples:

* shifts
* expenses
* vehicle checks
* incidents
* reports

---

## Local Helper

Used where a local desktop helper assists with reading driver cards or other local devices.

Important:

The helper must not be treated as legally certified C1B/DDD export unless this has been validated.

---

## External API

Future source for partner integrations.

Examples:

* GPS provider
* payroll software
* fuel card provider
* telematics platform

---

# Evidence Ownership

Every evidence item must be associated with ownership.

Ownership may include:

* user_id
* driver_id
* vehicle_id
* company_id
* import_id
* source_type
* uploaded_by

No evidence should exist without clear ownership.

---

# Evidence Status

Every evidence item should have a clear status.

| Status     | Meaning                             |
| ---------- | ----------------------------------- |
| received   | Upload received                     |
| validating | Validation in progress              |
| stored     | Original safely stored              |
| queued     | Waiting for processing              |
| processing | Processing underway                 |
| processed  | Successfully processed              |
| partial    | Partially processed                 |
| failed     | Processing failed                   |
| rejected   | Evidence rejected before processing |
| superseded | Replaced by newer evidence          |
| archived   | Retained but no longer active       |

---

# Validation Requirements

Validation should include:

* authenticated user
* permission to upload
* company scope
* allowed file type
* file size limit
* storage destination
* virus/malware scanning where available
* duplicate hash check
* metadata extraction
* safe filename handling

---

# File Integrity

Where possible, every uploaded file should receive:

* SHA-256 hash
* file size
* MIME type
* upload timestamp
* source label
* original filename
* storage path
* uploader identity

Hashing supports:

* duplicate detection
* audit traceability
* file integrity verification

---

# Duplicate Detection

Duplicate evidence should not create duplicate compliance results.

The system should detect duplicates using:

* file hash
* source metadata
* driver card number
* vehicle registration
* activity period
* upload timestamp
* import source

Duplicate handling should be cautious.

If uncertain, preserve the file but flag for review.

---

# Metadata Extraction

Metadata should be extracted before deeper processing where possible.

Examples:

Driver Card:

* card number
* driver name
* issuing country
* expiry date
* activity date range

Vehicle Unit:

* registration
* VU serial number
* calibration information
* download date
* date range

Documents:

* document type
* expiry date
* owner
* vehicle or driver link

Receipts:

* amount
* date
* vendor where available
* expense category

---

# Storage Requirements

Original evidence should be stored in private storage.

Storage should be:

* company-scoped
* permission-controlled
* immutable where practical
* backed up
* auditable

Original evidence must not be overwritten.

Derived files should be stored separately.

---

# Processing Queue

After evidence is stored, it may enter a processing queue.

The queue should support:

* retries
* failure logging
* progress tracking
* concurrency control
* prioritisation
* manual retry
* cancellation where appropriate

Long-running processes should never block the user interface unnecessarily.

---

# Normalisation

Different evidence sources should eventually be normalised into common platform structures.

Examples:

* timeline events
* driver activities
* vehicle activities
* compliance records
* reportable events
* evidence references

Normalisation allows Atlas and Reporting to consume consistent data regardless of source.

---

# Failure Handling

Failures must be safe.

If processing fails:

* original evidence remains stored
* status becomes failed or partial
* failure reason is recorded
* user receives understandable feedback
* retry is available where appropriate
* no duplicate partial records are created on retry

Failure messages should not expose sensitive internal details.

---

# Partial Imports

Partial imports are acceptable where useful.

Examples:

* identity decoded but activity missing
* metadata extracted but legal totals unavailable
* helper capture stored but not certified
* document stored but OCR unavailable

Partial imports must be clearly labelled.

Users should understand what is known and what is not known.

---

# Audit Events

The pipeline should create audit events for:

* upload started
* upload completed
* validation failed
* evidence stored
* duplicate detected
* processing started
* processing completed
* processing failed
* manual retry
* evidence exported
* evidence deleted or archived if permitted

Audit logs should include actor, timestamp and company scope.

---

# Security Requirements

The import pipeline is a high-risk area.

Security requirements include:

* authentication required
* role-based upload permissions
* company-scoped access
* private storage
* safe file handling
* file size limits
* content-type checks
* no executable processing
* no public access to sensitive files
* no secrets in upload metadata
* malware scanning roadmap

---

# User Experience Requirements

Users should see:

* upload progress
* validation status
* processing status
* success confirmation
* clear failure reason
* retry option
* link to resulting analysis
* whether the import is complete or partial

The interface should avoid technical noise for normal users.

Advanced diagnostics should be available only where useful.

---

# Retry Behaviour

Retry should be safe and idempotent.

A retry should not:

* duplicate activities
* duplicate infringements
* overwrite original evidence
* create inconsistent reports

Retries should use the existing evidence record where possible.

---

# Manual Fallback

Manual upload must remain available even if helper workflows improve.

This is important for:

* reliability
* customer support
* migration
* compatibility
* system outage resilience

Manual fallback should not be removed until replacement workflows are fully proven.

---

# Import Centre Role

The Import Centre should eventually become a clean operational workspace.

It should show:

* pending imports
* failed imports
* partial imports
* completed imports
* retry actions
* review actions

It should not overwhelm normal users with backend diagnostics.

Technical diagnostics should be collapsible or admin-only.

---

# Data Produced

The import pipeline may produce:

* evidence record
* metadata record
* processing run
* parsed activity
* timeline event
* compliance result
* warning
* infringement
* report trigger
* Atlas signal
* audit event

---

# Integration With Other Engines

## Timeline Engine

Receives normalised events.

## Compliance Engine

Receives timeline data and evidence references.

## Reporting Engine

Receives processed records and evidence links.

## Atlas

Receives structured findings, not raw unvalidated evidence.

---

# Definition of Done

The import pipeline is complete when:

* uploads are authenticated
* files are stored privately
* original evidence is preserved
* metadata is extracted
* status tracking works
* processing can be retried
* failures are clear
* duplicate handling exists
* audit events are created
* manual fallback works
* security review is complete
