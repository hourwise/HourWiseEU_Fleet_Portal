# Purpose

The Vehicle Unit Engine is responsible for importing, validating, decoding and normalising data obtained from digital tachograph Vehicle Units (VU).

Unlike the Driver Card Engine, which focuses on an individual driver's activity, the Vehicle Unit Engine focuses on the operational history of a vehicle.

It extracts trusted operational facts.

It does **not** calculate compliance.

---

# Responsibilities

The Vehicle Unit Engine is responsible for:

* importing VU files
* validating evidence
* decoding VU records
* extracting metadata
* extracting activities
* extracting events
* preserving original evidence
* publishing timeline events

It is **not** responsible for:

* legal calculations
* infringement detection
* reporting
* Atlas recommendations

# Purpose Within HourWise

Vehicle Unit data complements Driver Card data.

It provides information that may not exist on the driver's card and allows the platform to build a more complete operational picture.

Examples include:

* vehicle usage
* driver changes
* card insertion/removal events
* calibration information
* overspeed events
* faults
* vehicle activity history

---

# Metadata Extraction

Typical metadata includes:

* registration
* VIN where available
* VU serial number
* manufacturer
* calibration date
* calibration workshop
* download date
* recording period
* firmware or version information where available

Metadata should support searching and reporting.

---

# Vehicle Activity Extraction

The engine should extract:

* driving periods
* vehicle movement
* rest periods where represented
* card insertions
* card removals
* driver changes
* unknown activity
* manual entries where available

Activities should preserve their original timestamps.

---

# Event Extraction

Examples include:

* overspeed events
* power interruptions
* motion sensor events
* calibration events
* workshop activity
* authentication events
* faults
* security events
* card conflicts

Events should be preserved exactly as recorded.

Interpretation belongs to later processing stages.

---

# Driver Association

Vehicle Unit records may reference one or more drivers.

The engine should:

* associate driver card identifiers where possible
* preserve unknown driver references
* avoid guessing identities

Where confidence is low, relationships should remain unresolved until additional evidence becomes available.

---

# Calibration Records

Calibration information should be extracted separately.

Examples:

* calibration workshop
* calibration date
* next calibration due
* workshop card identifiers
* configuration changes

These records support compliance monitoring but are not themselves compliance decisions.

---

# Normalisation

Vehicle Unit records should be converted into standard HourWise event structures.

Examples:

```text
Vehicle Activity

Driver Change

Vehicle Event

Fault

Calibration

Security Event

Overspeed

Workshop Activity
```

The Timeline Engine should not need to know which file format produced the event.

---

# Original Evidence

Original Vehicle Unit files remain immutable.

Derived records should always reference:

* original upload
* processing run
* parser version

Nothing should overwrite original evidence.

---

# Duplicate Detection

Duplicate detection should consider:

* file hash
* VU serial number
* vehicle registration
* recording period
* download timestamp

Duplicate imports should never create duplicate timeline events.

---

# Error Handling

Errors should be categorised.

## Validation

Unsupported format.

Corrupt file.

Permission failure.

---

## Decode

Unexpected record.

Unknown structure.

Incomplete download.

Unsupported firmware.

---

## Storage

Repository unavailable.

Database failure.

Permission error.

---

## User

Duplicate upload.

Incorrect vehicle.

Unauthorised upload.

---

# Timeline Publication

The Vehicle Unit Engine publishes:

* activity events
* vehicle events
* metadata
* calibration records
* evidence references

It does not perform compliance calculations.

---

# Cross-Evidence Correlation

The Vehicle Unit Engine does not attempt to resolve every relationship itself.

Correlation is performed later by the Timeline Engine using information from:

* Driver Card Engine
* Driver App
* Vehicle Unit Engine
* Manual Records

This separation keeps the engine focused and deterministic.

---

# Performance Goals

The engine should:

* process large downloads reliably
* support historical archives
* recover safely from failure
* support reprocessing
* minimise unnecessary memory usage

Accuracy takes priority over speed.

---

# Security

Vehicle Unit data may contain commercially sensitive information.

Requirements include:

* authenticated uploads
* company-scoped access
* encrypted storage
* audit logging
* secure transport
* GDPR compliance where applicable

---

# Testing Requirements

Testing should include:

* valid VU downloads
* corrupted files
* duplicate uploads
* calibration records
* overspeed events
* multiple driver changes
* unsupported firmware versions
* parser regression tests
* timestamp validation

A growing library of anonymised and synthetic Vehicle Unit files should be maintained wherever legally possible.

---

# Parser Strategy

Parser implementation should remain isolated.

The engine exposes only normalised HourWise structures.

This allows parser libraries to change without affecting downstream systems.

---

# Explainability

Every published record should remain traceable to:

* original upload
* parser version
* processing run
* source record where available

This supports investigation, debugging and audit confidence.

---

# Engineering Rules

The Vehicle Unit Engine must never:

* calculate infringements
* modify original evidence
* invent missing driver identities
* discard unknown records silently
* change timestamps
* overwrite historical data

Unknown records should be preserved whenever practical.

---

# Future Enhancements

Potential developments include:

* remote VU download

* calibration trend analysis
* parser comparison mode
* confidence scoring
* vehicle health indicators
* enhanced security event analysis

# Relationship to Other Engines

## Driver Card Engine

Extracts driver-centred evidence.

## Vehicle Unit Engine

Extracts vehicle-centred evidence.

## Timeline Engine

Combines evidence into a unified chronological history.

## Compliance Engine

Applies legislation to the unified timeline.

## Atlas

Produces operational insight from validated compliance data.

Each engine has a distinct responsibility.

---

# Definition of Done

The Vehicle Unit Engine is complete when:

* files are validated
* metadata is extracted
* vehicle records are decoded
* events are decoded
* calibration information is preserved
* original evidence remains immutable
* timeline events are published
* duplicate detection works
* audit records are created
* parser regression tests pass
