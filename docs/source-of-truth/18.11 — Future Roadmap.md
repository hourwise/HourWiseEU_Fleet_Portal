## 1. Purpose

This document defines the future roadmap for the HourWise Compliance Intelligence Platform.

The roadmap exists to prevent scope creep during the first build while still preserving a clear path for future expansion.

The Compliance Intelligence Platform is intended to grow beyond basic tachograph import and reporting into a mature operational compliance system covering:

* driver card analysis
* vehicle unit analysis
* timeline reconstruction
* compliance outcomes
* evidence packs
* reporting
* audit preparation
* Atlas intelligence
* fleet risk monitoring
* maintenance and operational compliance links
* future integrations

This roadmap separates what must be built first from what should be added later.

---

## 2. Roadmap Principle

The core principle is:

> Build the trusted evidence foundation first. Add intelligence and automation only after the underlying records are reliable.

Atlas, dashboards, reports, alerts, and insights are only valuable if the system can trust:

* imported source files
* parser outputs
* timeline events
* compliance calculations
* evidence links
* audit records
* permissions
* report state

The future roadmap must not undermine evidential reliability.

---

## 3. Product Maturity Stages

The platform should evolve through clear maturity stages.

```text id="ne805e"
Stage 1: Import and Normalise
Stage 2: Analyse and Explain
Stage 3: Evidence and Reporting
Stage 4: Atlas-Assisted Review
Stage 5: Proactive Compliance Intelligence
Stage 6: Connected Fleet Compliance Platform
```

Each stage should be stable before the next stage depends on it.

---

## 4. Stage 1 — Import and Normalise

### 4.1 Goal

Build a reliable import pipeline for tachograph source data.

The system must be able to accept, process, validate, and store driver card and vehicle unit files.

### 4.2 Core Capabilities

* driver card file upload
* vehicle unit file upload
* import status tracking
* parser execution
* parser error handling
* raw file storage
* parsed data storage
* normalised activity creation
* duplicate detection
* import audit trail
* user/fleet ownership checks

### 4.3 Required Before Progression

Before moving beyond Stage 1, the system must reliably answer:

* Was the file received?
* Who uploaded it?
* What type of file is it?
* Was it parsed successfully?
* What driver, vehicle, or card does it relate to?
* What period does it cover?
* Was it already imported?
* What records were created?
* Were any parser errors found?

### 4.4 Deferred From Stage 1

Do not include at this stage:

* advanced AI interpretation
* automated enforcement-style language
* full report generation
* predictive compliance
* cross-fleet benchmarking
* external telematics integrations

---

## 5. Stage 2 — Analyse and Explain

### 5.1 Goal

Turn normalised activity data into meaningful timelines and compliance outcomes.

### 5.2 Core Capabilities

* timeline event creation
* activity grouping
* duty period detection
* rest period detection
* break detection
* daily totals
* weekly totals
* fortnightly totals
* driver/vehicle matching
* initial rule checks
* missing data detection
* confidence states
* compliance outcome records

### 5.3 Required Before Progression

Before moving beyond Stage 2, the system must reliably answer:

* What happened on a given day?
* Which activities were driving, work, rest, break, POA, or unknown?
* Where are the gaps?
* What data source supports each timeline event?
* Which compliance rules were checked?
* What outcome did each rule produce?
* Which outcomes are confirmed, possible, uncertain, or incomplete?

### 5.4 Deferred From Stage 2

Do not include at this stage:

* polished PDF reporting
* complex predictive analytics
* autonomous Atlas actions
* proactive notifications
* external regulator-style output
* payroll decisions based on compliance data

---

## 6. Stage 3 — Evidence and Reporting

### 6.1 Goal

Build the evidence layer that allows compliance outcomes to be reviewed, explained, and reported.

### 6.2 Core Capabilities

* evidence pack creation
* linking source files to outcomes
* linking timeline events to outcomes
* evidence completeness checks
* review notes
* report drafts
* report exports
* report status tracking
* audit trail for report generation
* manager review workflow

### 6.3 Required Before Progression

Before moving beyond Stage 3, the system must reliably answer:

* What evidence supports this outcome?
* What data is missing?
* Is this evidence pack complete?
* Who reviewed it?
* Has it been included in a report?
* Was the report exported?
* What exact records were used at export time?

### 6.4 Deferred From Stage 3

Do not include at this stage:

* automatic report sending
* autonomous sign-off
* AI-generated final conclusions without review
* driver notification workflows
* enforcement-ready legal wording
* automated dispute handling

---

## 7. Stage 4 — Atlas-Assisted Review

### 7.1 Goal

Add Atlas as a controlled interface over trusted platform records.

Atlas should help users understand and navigate the system, not replace the system’s deterministic engines.

### 7.2 Core Capabilities

* Ask Atlas interface
* context-aware prompts
* compliance outcome explanations
* evidence pack summaries
* report readiness summaries
* missing evidence explanations
* source-linked responses
* permission-aware answers
* audit logging
* draft review notes
* suggested next actions

### 7.3 Required Before Progression

Before moving beyond Stage 4, Atlas must reliably:

* respect permissions
* cite internal source records
* explain uncertainty
* avoid unsupported findings
* distinguish confirmed outcomes from possible outcomes
* avoid destructive actions
* log prompts and responses
* handle missing data safely

### 7.4 Deferred From Stage 4

Do not include at this stage:

* autonomous report export
* automatic manager sign-off
* automatic driver warnings
* predictive enforcement risk
* external email sending
* unrestricted natural language querying
* automatic compliance outcome modification

---

## 8. Stage 5 — Proactive Compliance Intelligence

### 8.1 Goal

Move from reactive review to proactive compliance monitoring.

The system should begin surfacing issues before they become operational problems.

### 8.2 Possible Capabilities

* daily compliance digest
* weekly risk summary
* missing download alerts
* repeated infringement pattern detection
* driver risk trend monitoring
* vehicle data gap alerts
* evidence pack completion reminders
* report deadline reminders
* manager task lists
* dashboard risk widgets
* configurable thresholds

### 8.3 Example Insights

Atlas or the dashboard may surface insights such as:

* “Driver card download overdue for 3 drivers.”
* “Vehicle unit data is missing for vehicle HW12 ABC.”
* “Two drivers have repeated reduced rest patterns this month.”
* “June compliance report has 4 unresolved evidence packs.”
* “One import failed and needs reprocessing.”
* “This vehicle has movement without matched driver card data.”

### 8.4 Required Safeguards

Proactive intelligence must:

* be configurable
* avoid alarmist language
* distinguish confirmed issues from possible issues
* avoid ranking drivers unfairly
* avoid automated disciplinary language
* link every insight to source records
* allow authorised users to mute or resolve alerts
* retain audit history

### 11.1 Capability IDs

Each major capability should receive a stable ID.

Example format:

```text id="v3o0xe"

CMP-001 (Drivers' Hours Rules)
INT-001 (Atlas Rule Engine)
REP-001 (Driver Daily Reports)
```

Capability IDs should allow requirements, tests, code modules, database tables, and UI components to cross-reference each other.

## 17. Multi-Jurisdiction Roadmap

HourWise may begin with UK/EU tachograph and working time logic, but the architecture should allow future jurisdiction expansion.

Possible future jurisdiction support:

* UK domestic rules
* EU rules
* AETR rules
* passenger transport variants
* light commercial vehicle tachograph obligations
* future regulatory changes
* non-European markets only if commercially justified

Rule logic should be versioned and configurable where possible.

Atlas and reports must always identify which rule set was applied.

## 18. Passenger Transport Roadmap

The platform may later support bus, coach, and passenger transport operators.

Possible adaptations:

* passenger transport duty patterns
* depot-based scheduling context
* route/service references
* different operational terminology
* school contract records
* coach hire records
* passenger transport compliance reports

The core tachograph import, timeline, evidence, and reporting architecture should remain reusable.

Passenger transport should not require a separate system unless the product, branding, or market positioning demands it.

## 19. Telematics and GPS Roadmap

Telematics integration is a future opportunity but should not be part of the first compliance MVP.

Possible future capabilities:

* vehicle location history
* movement confirmation
* vehicle utilisation
* route comparison
* mileage cross-checking
* movement without card alerts
* geofence events
* depot arrival/departure detection
* integration with smaller GPS providers

Important constraints:

* provider APIs vary
* commercial agreements may be needed
* location data is sensitive
* driver consent and device ownership matter
* company-device and personal-device rules differ
* GPS should support evidence but not replace tachograph records

Telematics should be treated as supplementary evidence, not the legal source of tachograph truth.

## 24. Risk Register

The roadmap should track major product and technical risks.

### 24.1 Tachograph Parser Risk

Risk:

* parser output may be incomplete, incorrect, or unsupported for some files

Mitigation:

* parser versioning
* test fixtures
* import diagnostics
* manual review states
* clear unsupported-file handling

### 24.2 Legal Language Risk

Risk:

* system may overstate compliance findings

Mitigation:

* cautious wording
* confidence states
* evidence-backed outcomes
* manager review workflow
* legal disclaimer copy
* no unsupported Atlas claims

### 24.3 AI Hallucination Risk

Risk:

* Atlas may produce unsupported explanations

Mitigation:

* retrieval-first design
* source cards
* restricted context
* response categories
* confidence states
* prompt injection protection
* audit logging

### 24.4 Data Security Risk

Risk:

* fleet, driver, or vehicle data could be exposed incorrectly

Mitigation:

* tenant isolation
* role permissions
* access logs
* export controls
* secure storage
* support access controls

### 24.5 Scope Creep Risk

Risk:

* platform becomes too broad before the compliance foundation is stable

Mitigation:

* roadmap stages
* MVP boundary
* clear deferred features
* separate future specifications
* phased build checklist

## 30. Summary

The HourWise Compliance Intelligence Platform should grow in controlled stages.

The first priority is not AI, dashboards, or integrations.

The first priority is trust.

Trust comes from:

* reliable imports
* clear parser outputs
* accurate timelines
* cautious compliance outcomes
* complete evidence packs
* auditable reports
* permission-aware interfaces
* transparent explanations

Atlas, proactive intelligence, telematics, mobile workflows, and wider fleet compliance features should build on that foundation.

The long-term vision is a connected fleet compliance platform that helps operators understand risk, prepare evidence, manage reports, and run safer, better-documented fleets.

But the correct build order is clear:

> Import first.
> Analyse second.
> Evidence third.
> Report fourth.
> Assist fifth.
> Automate only when the foundation is trustworthy.
