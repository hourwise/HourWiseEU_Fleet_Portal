# Purpose

This document defines the internal architecture of the Compliance Intelligence Platform.

The architecture is designed around **evidence**, not file types.

Every component exists to transform operational evidence into trusted compliance intelligence while preserving traceability, explainability and auditability.

---

# Architectural Philosophy

The platform should not think in terms of:

* DDD files
* C1B files
* Driver App records
* GPS

Instead it should think in terms of:

**Evidence**

Every evidence source enters the platform through the same logical pipeline.

This allows future evidence sources to be added without redesigning the system.

# Core Architecture Layers

The platform consists of nine logical layers.

---

# Layer 1 — Evidence Sources

Purpose

Collect operational evidence.

Current sources include:

* Driver Card
* Vehicle Unit
* Driver App
* Manual records

Future sources include:

* GPS
* Telematics
* OCR
* DVSA integrations
* Workshop systems

This layer should remain open for future expansion.

---

# Layer 2 — Import Pipeline

Purpose

Receive evidence safely.

Responsibilities

* upload
* validation
* virus scanning (future)
* integrity checking
* metadata extraction
* duplicate detection

The pipeline must never modify original evidence.

---

# Layer 3 — Validation Layer

Purpose

Verify evidence quality before processing.

Checks include:

* supported format
* checksum (where applicable)
* completeness
* ownership
* duplicate imports
* corruption detection

Invalid evidence should never enter processing.

---

# Layer 4 — Evidence Repository

Purpose

Store all evidence safely.

The repository contains:

Original files

Derived metadata

Import history

Audit information

Processing status

Evidence relationships

The repository is immutable.

Original evidence should never be altered.

---

# Layer 5 — Timeline Engine

Purpose

Convert evidence into a chronological operational timeline.

Examples:

Driving

Other Work

Break

POA

Rest

Events

Defects

Incidents

Vehicle checks

Documents

Future GPS events

Everything becomes a timeline event.

This is one of the defining architectural concepts of HourWise.

---

# Layer 6 — Compliance Engine

Purpose

Interpret timeline events.

Responsibilities

Drivers' Hours

Working Time

Rule evaluation

Infringement detection

Warnings

Risk scoring

Every calculation should be reproducible.

---

# Layer 7 — Intelligence Layer

Purpose

Transform compliance results into operational insight.

Examples

Atlas Briefing

Risk Scores

Recommendations

Future Predictions

Trend Analysis

This layer never replaces evidence.

It explains it.

---

# Layer 8 — Reporting Engine

Purpose

Generate trusted outputs.

Outputs include:

PDF

CSV

Driver reports

Fleet reports

Audit packs

Evidence exports

Reports should always reference supporting evidence.

---

# Layer 9 — Platform Consumers

Information is consumed by:

Fleet Portal

Driver App

Atlas

Resource Centre

Future APIs

Every consumer should retrieve processed information rather than analysing evidence independently.

---

# Design Rules

Every module should:

Have one responsibility.

Be independently testable.

Produce deterministic outputs.

Avoid duplicate logic.

Remain replaceable without affecting other modules.

---

# Separation of Responsibilities

## Evidence Layer

Stores facts.

Never interprets them.

---

## Timeline Layer

Organises facts.

Never judges them.

---

## Compliance Layer

Applies legislation.

Never changes evidence.

---

## Intelligence Layer

Explains results.

Never alters calculations.

---

## Reporting Layer

Communicates outcomes.

Never performs compliance calculations.

---

# Processing Principles

Processing should always be:

Deterministic

Repeatable

Auditable

Recoverable

Versioned

Traceable

Explainable

---

# Traceability

Every report should be traceable back through every processing stage.

```text
PDF Report

↓

Compliance Result

↓

Timeline Event

↓

Evidence Item

↓

Original Upload
```

Nothing should become disconnected.

---

# Failure Recovery

If processing fails:

Original evidence remains.

Failure reason recorded.

Processing can be retried.

No evidence is lost.

Partial processing is clearly identified.

---

# Scalability

The architecture should support future expansion without redesign.

Future additions include:

Remote tachograph download

Live telematics

AI anomaly detection

Driver behaviour analysis

Wearables

Digital inspections

Partner integrations

All should enter through the same evidence pipeline.

---

# Architectural Principles

The architecture prioritises:

Evidence preservation

Transparency

Explainability

Security

Traceability

Maintainability

Scalability

Long-term extensibility

---

# Definition of Success

The system architecture succeeds when:

New evidence sources can be added without redesign.

Compliance calculations remain trustworthy.

Every report is explainable.

Every audit trail is complete.

Atlas receives structured, reliable data.

No evidence is lost or altered.
