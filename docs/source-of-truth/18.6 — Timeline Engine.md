# Purpose

The Timeline Engine is responsible for transforming evidence from multiple independent sources into a single chronological operational history.

It is the central organising component of the Compliance Intelligence Platform.

Unlike the Driver Card Engine or Vehicle Unit Engine, the Timeline Engine does not decode files.

Unlike the Compliance Engine, it does not apply legislation.

Its sole responsibility is to organise trusted evidence into a coherent sequence of events.

---

# Philosophy

Every operational fact becomes a timeline event.

Regardless of its origin.

Examples include:

* Driver Card activity
* Vehicle Unit events
* Driver App actions
* Vehicle checks
* Incident reports
* Expenses
* Messages
* Documents
* GPS events (future)
* Telematics (future)

Everything is simply another event in time.

---

# Architectural Position

```text
Evidence Sources

↓

Evidence Repository

↓

Timeline Engine

↓

Compliance Engine

↓

Atlas

↓

Reporting
```

The Timeline Engine sits between evidence extraction and compliance interpretation.

---

# Core Responsibilities

The Timeline Engine is responsible for:

* ordering events chronologically
* correlating evidence
* preserving evidence references
* identifying gaps
* identifying overlaps
* normalising timestamps
* publishing unified timelines

It is **not** responsible for:

* infringement detection
* legal interpretation
* recommendations
* reporting

---

# Timeline Philosophy

The Timeline is a factual representation of events.

It should answer one question:

> **"What happened, and when?"**

Nothing more.

Interpretation happens later.

---

# Timeline Event Model

Every event should contain a common structure.

Example:

```text
Event ID

Timestamp Start

Timestamp End (optional)

Event Type

Evidence Source

Related Driver

Related Vehicle

Location (optional)

Confidence

Evidence Reference

Metadata

Processing Version
```

All events should follow the same schema regardless of source.

# Evidence Correlation

The Timeline Engine combines evidence from multiple sources.

Examples:

Driver Card

*

Vehicle Unit

*

Driver App

↓

Unified Timeline

The engine should preserve every source independently while presenting one operational history.

---

# Timestamp Handling

The Timeline Engine must preserve original timestamps.

Where conversion is required:

* original value retained
* normalised value stored
* timezone recorded
* conversion method documented

No timestamp should be silently altered.

---

# Event Ordering

Ordering priority:

1. Original timestamp
2. Evidence confidence
3. Import sequence where necessary

Ordering rules must be deterministic.

---

# Evidence References

Every timeline event should reference:

* original upload
* parser version
* evidence ID
* processing run
* import source

Nothing should become disconnected from its origin.

---

# Confidence Levels

Timeline events may carry confidence indicators.

Examples:

Confirmed

Verified

Imported

Manual

Estimated

Unknown

Confidence never changes the evidence.

It helps users understand provenance.

---

# Gap Detection

The Timeline Engine should identify periods where expected evidence is missing.

Examples:

* missing driver activity
* unexplained vehicle movement
* incomplete download
* missing app data

Gap detection identifies missing evidence.

It does not infer what happened.

---

# Overlap Detection

Examples:

Two simultaneous driving periods.

Conflicting activity records.

Multiple active drivers.

Duplicate evidence.

The Timeline Engine identifies conflicts.

It does not resolve them automatically.

---

# Cross-Source Linking

Examples:

Driver App shift

linked to

Driver Card driving period

linked to

Vehicle Unit movement

linked to

Expense

linked to

Incident

These relationships enable richer reporting and Atlas intelligence.

---

# Unknown Events

Unknown or unsupported events should still appear in the timeline.

They should be labelled appropriately.

Unknown information should never disappear simply because it is not yet understood.

---

# Timeline Publication

The Timeline Engine publishes:

Unified Timeline

Evidence Links

Event Relationships

Gap Indicators

Conflict Indicators

Confidence Levels

Compliance Engine consumes this output.

---

# Timeline Storage

Timeline events should remain immutable after publication.

If recalculation is required:

* create a new processing version
* retain previous versions where appropriate
* preserve audit history

---

# Performance Goals

The Timeline Engine should support:

* large fleets
* many years of history
* fast chronological queries
* efficient filtering
* deterministic rebuilding

Scalability is a primary design goal.

---

# Timeline Queries

Users should eventually be able to view timelines by:

Driver

Vehicle

Company

Date

Incident

Evidence Source

Compliance Issue

Future custom filters

---

# Security

Timeline visibility depends upon:

authentication

authorisation

company ownership

user role

Evidence should never become visible simply because it exists within a timeline.

---

# Explainability

Every event should answer:

Where did this come from?

What evidence supports it?

Which processing version created it?

Can the original evidence be viewed?

---

# Engineering Rules

The Timeline Engine must never:

calculate infringements

change original evidence

invent missing events

guess timestamps

delete unsupported records

apply legislation

Its responsibility is chronological truth.

---

# Future Enhancements

# Definition of Done

The Timeline Engine is complete when:

events are published consistently

ordering is deterministic

evidence references exist

timestamps are preserved

gaps are identified

conflicts are identified

cross-source relationships work

unknown events are retained

audit history is preserved

performance targets are met
